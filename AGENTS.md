# AGENTS.md

## 文档目的

本文件用于为后续参与本项目的 AI 助手与开发者提供统一约束，确保对话、文档、代码实现、UI 风格和迭代流程保持一致。

本项目是一个基于 `Wails + Go + React + TypeScript` 的跨平台桌面端工具，用于增强公司内部 Redmine 的使用体验。

## 语言与沟通规范

- 所有对话默认使用简体中文
- 所有 Markdown 文档默认使用简体中文
- 所有代码注释默认使用简体中文
- 如无明确必要，不切换为英文回复
- 代码标识符、第三方库名称、CLI 命令、HTTP 字段名、日志关键字可保留原始语言
- 面向用户的文案、页面标题、字段说明、空态文案、错误提示优先使用简体中文

## 编码与文件格式规范

- 所有新建或修改文件必须使用 `UTF-8` 编码
- 严禁提交乱码、GBK、ANSI 或不可读字符
- 修改文档或代码时，需检查中文是否正常显示
- 文档文件优先使用 `.md`

## 注释规范

- 代码中需要的地方必须补充详细注释，注释使用简体中文
- 对以下场景必须写清楚注释：
- Redmine API 字段映射
- 自定义字段兼容逻辑
- 统计聚合计算逻辑
- 主题切换与 Token 映射逻辑
- Wails 前后端绑定边界
- 权限、兼容性、降级处理
- 非直观的状态流转逻辑
- 非常直白、无需解释的一行赋值或样板代码不要堆砌无意义注释
- 注释目标是帮助后续维护者理解“为什么这样做”和“这里依赖什么约束”，不是重复代码表面含义

## 项目固定技术栈

除非产品需求明确改变，否则本项目固定采用以下技术栈：

- 桌面框架：`Wails`
- 后端语言：`Go`
- 前端框架：`React`
- 前端语言：`TypeScript`
- 构建工具：`Vite`
- 样式系统：`Tailwind CSS`
- 组件体系：`shadcn/ui + Base UI`
- 状态管理：`Zustand`
- 数据请求与缓存：`TanStack Query`

不引入以下方案：

- `Ant Design`
- `Electron`
- 独立业务后端服务
- 与当前设计体系冲突的 UI 框架默认样式

## UI 与设计约束

- 页面布局、功能流、层级结构必须严格参考 `ui/stitch_1.html` 到 `ui/stitch_7.html`
- **配色系统必须严格遵循设计稿中的 The Monolith (Graphite Dark) 体系**，例如主色调为 `#bdc2ff`，深色背景为 `#121315` 等。
- 严禁擅自将配色改为纯黑/纯白的 Vercel/Linear 单色风格。
- 在保留原版配色的基础上，强化玻璃态、光影和深度边缘等高级质感。
- **不再拘泥于默认 MD3 的 Token 系统或者纯扁平设计**，杜绝大面积僵硬边框、干瘪的组件和简单的色块堆积
- 必须支持 `明 / 暗 / 跟随系统` 三种主题模式
- 切换主题时，不允许改变布局结构、字号层级、信息密度与交互节奏
- 坚决不允许擅自改成通用平庸的传统后台管理系统风格
- 所有新页面、新组件必须与现有深度定制过的高质感（高光、细致阴影、精致圆角等）Token 体系保持一致

## UI 质感与组件使用要求

- **开发时必须优先使用 shadcn/ui 组件库**。切勿自行使用原生 HTML/Tailwind 手捏比如 `textarea`、头像 (`Avatar`)、滚动条或弹层等通用交互组件。如果还没安装，请运行 `npx shadcn@latest add <组件>`。(注: Windows 下如果遇到缓存报错请加上 `--cache ./.npm-cache` 参数)。只有当 shadcn 体系彻底无法支持时，才允许自行封装原生 HTML 覆盖。
- 页面级布局必须优先还原 `ui/stitch_1.html` 到 `ui/stitch_7.html` 的真实结构，不允许长期保留“临时骨架”“占位壳层”“为方便开发而改成顶部菜单”的过渡实现
- 对主流界面元素，必须复用并基于 `shadcn/ui` 进一步打磨（注入阴影发光特质），不要使用默认的低质感平庸样式
- 只有极其确信的情况下才引入新的第三方界面库，现有组件的样式调优目标是：创造极为克制却具有极强工艺质感的 UI（细腻的 hover 过渡、克制使用实线边框、善借暗角与内发光）
- 不允许为了赶进度牺牲页面质感：即使是迭代中的页面，也必须先把整体布局骨架、层级光影、留白呼吸感、表面高光层次做对，再继续填业务内容
- 最终代码必须使用项目深度定制后的 `shadcn/ui + Base UI + Tailwind` 体系去还原甚至超越原设计稿的细节质感，坚决不允许复用低劣的页面壳层或默认组件形态

## 产品页面流约束

- 桌面端启动后，必须先判断当前是否已有可用的 Redmine 配置
- 如果没有配置，应用首页必须直接显示 `stitch_1` 对应的连接/配置页面，而不是先进入带侧栏的系统壳层
- 如果已有配置，应用必须先基于该配置做接口校验或初始化刷新；校验通过后，才进入 `stitch_2` 到 `stitch_7` 对应的系统内页面
- `stitch_2` 到 `stitch_7` 才是进入系统后的页面集合；`stitch_1` 不属于系统内主壳层页面
- 已进入系统后，如需切换 Redmine 配置，入口应放在“设置”页面中处理，而不是继续在任务中心或其他业务页顶部长期放置连接表单

## 开发前必须阅读的文档

开始开发任何模块前，必须先阅读以下文档：

- `docs/redmine-pro-full-requirements.md`
- `docs/development-plan.md`

要求：

- 先确认当前任务属于哪个 Phase
- 先确认该功能是否在需求范围内
- 先确认对应的 UI 稿件是哪一张
- 先确认 Redmine API 是否直接支持，还是需要聚合或兼容处理

## 迭代开发规则

- 每次开发只处理一个明确、最小可交付的子任务
- 开发完成后，必须更新 `docs/development-plan.md` 中对应任务的勾选状态
- 如果实现过程中发现需求、设计、技术边界发生变化，必须同步更新相关文档
- 不允许代码已经改动，但文档仍停留在旧状态
- 不允许一次性大范围无计划重构
- 如果只是为了满足当前模块，不要提前引入未来阶段才需要的复杂架构

## 代码组织原则

- 前端按业务模块组织，避免把所有页面逻辑堆在 `App.tsx`
- Go 侧按 API 域与统计域拆分，避免长期把所有逻辑堆在 `app.go`
- 公共能力必须抽取复用，避免重复实现筛选器、状态标签、工时表单、布局壳层等能力
- 页面层不直接写大段业务逻辑，应通过服务层、hook、store 或 Go 绑定方法承载
- 不要为了“看起来高级”而引入无收益的抽象层

## Redmine API 约束

- 默认兼容基线必须以公司当前 Redmine 实例为准，即页面页脚显示的 `Powered by Redmine © 2006-2022 Jean-Philippe Lang`
- 实现功能时，不允许想当然假设更高版本 Redmine 的新接口、新字段、新行为一定存在
- 所有 Redmine 数据访问统一通过 Wails 的 Go 侧处理
- 前端不直接请求 Redmine HTTP API
- 所有 API 兼容性差异必须在代码中有明确注释
- 对 `S2`、`S3`、`S4` 类需求，必须严格参考需求文档中的能力标记
- 如果某个功能依赖实例中的自定义字段，必须在实现和注释中写明依赖关系
- 如果标准 Redmine API 不能直接满足需求，允许通过 Go 侧二次聚合、分页拉取、字段映射、统计计算等方式实现
- 无论是直接调用 Redmine API，还是在 Go 侧做二次聚合开发，最终交付都必须以“功能能在该版本 Redmine 上真正跑通”为标准，而不是只完成静态页面、伪按钮或理论设计
- 如果某项需求在该版本 Redmine 上无法可靠实现，必须明确标注阻塞原因、版本限制和替代方案，不能假装已支持

## 文档维护规则

- 新增模块、页面、接口、重要字段映射时，优先更新文档再推进实现
- Markdown 文档结构要清晰，标题层级稳定，便于 AI 多轮续写
- 文档中的任务状态、实际代码状态、实现边界必须一致
- 如果发现历史文档内容不准确，应直接修正，不保留模糊表述

## 质量要求

- 修改代码后，优先保证能通过当前构建
- 前端改动至少应保证 `npm run build` 可通过
- Go 改动至少应保证 `go build ./...` 可通过
- 涉及复杂逻辑时，优先补测试或最少补充清晰注释
- 新功能必须处理加载态、空态、错误态

## 默认工作方式

后续 AI 助手在本项目中工作时，默认遵循以下顺序：

1. 阅读需求文档与开发规划文档
2. 识别当前最小未完成任务
3. 对照 UI 稿件实现页面或能力
4. 接入真实 Redmine API 或明确标注占位实现
5. 补充必要的简体中文注释
6. 更新文档状态
7. 再进入下一轮开发

## 补充说明

- 如果用户明确提出新的产品需求，应同步补到需求文档或开发规划文档
- 如果实现与设计稿存在冲突，优先与用户确认，而不是自行改风格
- 如果发现仓库存在自己未产生的异常改动，应暂停并先确认处理方式
