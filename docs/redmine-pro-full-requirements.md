# Redmine Pro 全功能需求清单

更新时间：2026-03-23

## 1. 文档目标

本文档用于整理 `redmine-pro` 桌面端的完整功能需求，作为后续产品确认、页面拆分、接口设计和迭代计划的基础。

项目背景：

- 当前公司内部使用的 Redmine 页面老旧，交互效率低
- 常见痛点包括：筛选能力弱、人员选择依赖长下拉、项目/版本查找不便、问题处理路径长、统计报表能力不足
- 目标是基于 `Wails + React + TypeScript + Tailwind CSS + shadcn/ui + Base UI + TanStack Query + Zustand` 构建跨平台桌面端工具
- 数据来源以 Redmine 标准 REST API 为主，不预设额外服务端

## 2. 需求范围依据

本需求清单来自以下输入：

- `ui/stitch_1.html` 到 `ui/stitch_7.html` 的 UI 稿件
- 用户当前高频使用的 6 类业务场景
- Redmine 官方 REST API 文档能力边界

## 3. 能力标记说明

- `S1`：标准 Redmine API 直接支持
- `S2`：标准 API 支持，但需要客户端聚合、缓存、编排或二次计算
- `S3`：依赖 Redmine 实例已有字段、角色权限或配置
- `S4`：标准 Redmine API 不直接支持，需插件、定制接口或二期处理

## 4. 版本兼容前提

当前页面页脚显示为 `Powered by Redmine © 2006-2022`，无法仅凭页脚精确判断子版本，因此需求设计需要考虑兼容性差异：

- `/my/account` 为 Redmine `4.1+`
- `project include=issue_custom_fields` 为 Redmine `4.2+`
- issue 详情中的 `allowed_statuses` 为 Redmine `5.0.x+`

因此桌面端应按“能力探测 + 自动降级”实现，而不是将新接口作为唯一依赖。

## 5. 产品目标

`redmine-pro` 的目标不是简单复刻 Web 版 Redmine，而是解决以下使用痛点：

- 让开发人员更快查看和处理“我的问题”
- 让问题详情页支持高频编辑操作的一站式完成
- 让工时登记、日报、月报更直观
- 让项目负责人和研发组长能按项目、版本、人员、问题类型、工时等维度完成统计
- 让项目、版本、成员、问题的定位和筛选比原版更快、更舒适

## 6. 全功能清单

### 6.1 连接与账户

- 配置 Redmine 地址与 API Key，并测试连接状态。`S1`
- 读取当前登录用户信息，显示姓名、登录名、邮箱等基础资料。`S1`
- 支持多个 Redmine 实例配置，本地切换。`S2`
- 本地加密保存 API Key。`S2`
- 启动时检测当前实例的 API 能力和权限范围。`S2`
- 根据实例能力决定是否启用增强功能，例如 `allowed_statuses`、自定义字段元数据读取。`S2`

- 我负责的问题总数。`S2`
- 今日到期问题数。`S2`
- 已逾期问题数。`S2`
- 今日登记工时。`S2`
- 最近更新问题。`S2`
- 支持按项目、版本、状态、优先级、跟踪类型、分类、更新时间快速筛选。`S1 + S2`
- 支持最近访问、收藏项目、收藏版本、固定筛选视图。`S2`
- 支持工作台快速动作：
- 开始处理
- 转测试
- 解决问题
- 重新指派
- 登记工时

- 不将“创建/编辑 Redmine Query”纳入一期承诺，因为标准 Queries API 仅支持读取。`S4`

## 7. 建议统一的业务字段口径

为了避免后续统计口径混乱，建议现在就固定字段语义：

- `指派给`：直接使用 Redmine 原生 `assigned_to_id`
- `责任人`：优先复用你们实例中已有自定义字段；若没有，不建议一期新增一个与 `assigned_to` 语义冲突的字段
- `原因分类 / 错误类型`：优先映射到 `问题分类` 或列表型 `custom field`
- `功能 / 错误 / 优化`：优先映射到 `tracker`
- `设计 / 测试 / case`：优先映射到 `time entry activity` 或自定义字段
- `版本 / 迭代`：优先使用 Redmine `fixed_version`
- `部门 / 小组`：若 Redmine 用户资料中没有相关字段，则需通过 group 或本地维护团队映射实现

## 8. Redmine 标准 API 边界

以下事项不应在一期中做强承诺：

- 标准 API 下，普通用户通常无法调用全局 `/users` 列表做全局人员搜索
- 标准 Queries API 支持读取，但不支持完整的创建、编辑查询管理能力
- 标准 API 没有真正的批量更新问题接口，批处理只能通过客户端逐条请求实现
- “敏感操作日志”“审批流”“组织架构”“部门绩效规则”等不是 Redmine 标准 REST API 自带能力
- 高级管理报表不是 Redmine 现成接口返回，需要桌面端基于问题、工时、成员、版本、历史记录做聚合计算

## 9. 对原版 Redmine 痛点的针对性改进

### 9.1 查找效率

- 所有下拉改为可搜索选择
- 支持项目、版本、成员、问题号的统一搜索入口
- 支持最近使用、收藏、固定视图

### 9.2 处理效率

- 问题列表与详情同屏展示，减少页面跳转
- 常用操作前置，例如解决、转测试、重开、指派、登记工时
- 支持一键完成任务和一键转测试

### 9.3 筛选效率

- 支持多条件组合筛选
- 支持保存本地筛选方案
- 支持恢复上次工作上下文

### 9.4 统计与管理

- 按项目、版本、人员、状态、跟踪类型、时间范围生成统计
- 支持版本维度的工时、问题、质量分析
- 支持组长查看团队工作饱和度和问题分布
